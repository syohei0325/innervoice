## 7つの堀（詳細）

### 1. データモート（Data Moat）

**定義**：実行データの量と質が競合参入を困難にする

**指標**：
- **確定データ件数**：≥1,000万件/年
- **Frontier Ratio**：≥35%（電話/非APIでの確定割合）
- **テンプレ種類**：≥100種類（病院/飲食/再配達/美容/不動産...）

**なぜ堀になるか**：
- 実行データは「どの言い方で成功するか」の知見の宝庫
- Frontier（電話しかない島）の攻略ノウハウは真似できない
- テンプレが増えるほど新規参入のハードルが上がる

**実行計画**：
- **0-3m**：通話テンプレ3本で月間1,000件確定
- **3-6m**：テンプレ10本で月間10,000件確定
- **6-12m**：テンプレ50本で月間100,000件確定（年間120万件）
- **12-18m**：テンプレ100本で月間1,000,000件確定（年間1,200万件）

---

### 2. 埋め込み（Embedding）

**定義**：顧客の業務フローに深く組み込まれ、スイッチングコストが高い

**指標**：
- **台帳SaaS有償契約**：≥1,000社
- **監査エクスポート利用**：≥3,000/月
- **平均契約期間**：≥24ヶ月
- **解約率（Churn）**：<5%/年

**なぜ堀になるか**：
- 台帳（Execution Ledger）は監査・eDiscovery・コンプライアンスに必須
- 一度組み込まれると「外せない」（業務が止まる）
- 他社に乗り換えると過去の監査証跡が失われる

**実行計画**：
- **0-3m**：台帳SaaS PoC 3社（無償）
- **3-6m**：有償化開始、50社獲得（¥20/席/月）
- **6-12m**：500社獲得、エンタープライズプラン追加
- **12-18m**：1,000社獲得、解約率<5%/年達成

---

### 3. 生態系（Ecosystem）

**定義**：外部プロバイダ・開発者・入口が増え、ネットワーク効果が働く

**指標**：
- **Provider認定**：≥50社（Twilio/Telnyx/Vonage/Plivo...）
- **外部入口からの`/confirm`**：≥3億/月（ChatGPT/Claude/Gemini/検索...）
- **MCPツール採用**：≥10社のLLMクライアント
- **コネクタ市場**：≥200種類の認定コネクタ

**なぜ堀になるか**：
- Providerが増えるほど冗長性・品質・価格競争が働く
- 入口が増えるほど「Yohaku = 実行の標準」になる
- 開発者が集まると「Yohaku互換」がデファクトスタンダードに

**実行計画**：
- **0-3m**：Provider 2社（Twilio/Telnyx）でPoC
- **3-6m**：Provider認定プログラム開始、15社獲得
- **6-12m**：外部入口5本（ChatGPT Apps/Claude/自社App/検索/OS統合）
- **12-18m**：Provider 50社、外部`/confirm`≥3億/月

---

### 4. COGS（Cost of Goods Sold：原価率）

**定義**：粗利率が高く、スケールするほど利益が出る

**指標**：
- **粗利率**：≥70%
- **SLA**：99.9%（月間ダウンタイム<43分）
- **平均レイテンシ**：`/v1/plan` p50≤700ms
- **原価内訳**：LLM 20% / 通話 15% / インフラ 10% / その他 5%

**なぜ堀になるか**：
- 粗利70%は競合が価格競争を仕掛けても耐えられる
- SLA 99.9%はエンタープライズ採用の必須条件
- 原価最適化（Model Routing Layer）で競合より安く提供できる

**実行計画**：
- **0-3m**：原価計測開始、Model Routing Layer実装
- **3-6m**：粗利50%達成、SLA 99.5%
- **6-12m**：粗利60%達成、SLA 99.7%
- **12-18m**：粗利70%達成、SLA 99.9%

---

### 5. カウンターポジショニング（Counter-Positioning）

**定義**：既存大手が真似できないビジネスモデル

**指標**：
- **成果課金比率**：売上の≥50%が$0.03/アクション
- **台帳SaaS比率**：売上の≥30%が$20/席/月
- **サブスク比率**：売上の<20%（B2C個人向け）

**なぜ堀になるか**：
- Google/Apple/Microsoftは「成果課金」に移行できない（既存売上を食う）
- 大手は「透明性（AXI公開）」をやりたがらない（品質が露呈）
- 台帳SaaSは「規制対応」が必要で大手は参入しにくい

**実行計画**：
- **0-3m**：成果課金モデル設計（$0.03/act）
- **3-6m**：台帳SaaS開始（$20/席/月）、売上比率20%
- **6-12m**：成果課金50%、台帳30%、サブスク20%
- **12-18m**：成果課金60%、台帳20%、サブスク20%

---

### 6. 透明性（Transparency）

**定義**：品質指標を公開し続け、信頼を獲得

**指標**：
- **AXI連続公開**：52週（1年間）
- **Security KPI連続公開**：52週
- **クレーム率**：<0.5%/月
- **NPS**：≥50

**なぜ堀になるか**：
- 透明性は「信頼」を生む（B2B顧客は品質を重視）
- 52週連続公開は「継続性」の証明（一時的ではない）
- 競合は真似しにくい（品質が低いと公開できない）

**実行計画**：
- **0-3m**：AXI週次公開開始（Week 1-12）
- **3-6m**：Security KPI追加（Week 13-24）
- **6-12m**：AXI Leaderboard公開（Week 25-48）
- **12-18m**：連続52週達成（Week 49-52+）

## 決定打（Moatをロック）

### 1. PoEx（Proof-of-Execution：実行証明）

**概要**：
- すべての実行に「改ざん不可能な証明」を発行
- 週次でMerkle Rootを透明性ログに掲示
- 第三者が検証可能

**実装**：
```json
{
  "receipt_id": "rcp_abc123",
  "approve_id": "aprv_abc123",
  "plan_id": "pl1",
  "actions": [{"action":"calendar.create","id":"evt_123","status":"ok"}],
  "ts": "2025-10-19T10:10:02Z",
  "device_sig": "base64...",
  "server_sig": "base64...",
  "merkle_root_week": "2025-W45"
}
```

**なぜ決定打か**：
- B2B顧客は「監査証跡」を重視
- 改ざん不可能な証明は競合が真似しにくい
- 規制対応（eDiscovery）に必須

---

### 2. Conformance Test + Yohaku-Compatible バッジ

**概要**：
- Provider/コネクタの仕様準拠を自動テスト
- 合格先に「Yohaku-Compatible」バッジ付与
- AXI Leaderboardに掲載

**実装**：
```bash
# Conformance Test Suite
$ yohaku-test-suite --provider twilio
✓ 署名検証（HMAC）
✓ 冪等性（24h）
✓ 発信者番号ポリシー
✓ 録音同意（地域差）
✓ インシデント通知（72h）
✓ データ地域固定
✓ ConfirmOS準拠

Result: PASS
Badge: Yohaku-Compatible (Gold)
```

**なぜ決定打か**：
- 品質保証の自動化（手動審査は限界）
- バッジが「信頼の証」になる
- 生態系の品質を担保

---

### 3. AXI Treaty（公開契約）

**概要**：
- AXI指標の定義・閾値・返金ルールを公開契約化
- SLA違反時は自動返金（例：通話成功率<85%で10%返金）
- 透明性を「契約」に昇華

**実装**：
```yaml
# AXI Treaty v1.0
sla:
  call_success_rate:
    threshold: 0.90
    penalty: 10% refund if < 0.85
  misexec_rate:
    threshold: 0.005
    penalty: 20% refund if > 0.01
  ttc_p50:
    threshold: 1500ms
    penalty: 5% refund if > 2000ms
```

**なぜ決定打か**：
- 透明性を「契約」にすると信頼が最大化
- 競合は真似しにくい（品質が低いと返金地獄）
- エンタープライズ顧客は「保証」を重視
